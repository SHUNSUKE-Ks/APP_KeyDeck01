# KeyDeck 構想と自走開発体制 v1（正）

作成: FABLE 2026-07-20。これは本プロジェクトの北極星。全AIはCLAUDE.mdの次にこれを読む。

## 1. ゴールの言語化（North Star）

**「あらゆるアプリ・あらゆる画面に、最適な入力デバイスを後付けできる基盤」**

- Hub（Rust）が頭脳、端末はブラウザを開くだけの「ただのボタンの板」。この構図は不変
- キーボード・Inspector・Deck・マクロ等の**入力面（Surface）はすべてJSONデータ**であり、コードを書かずに増やせる・変えられる・配れる
- 最終形: 「このアプリのこの画面には、この入力面が合う」をAIが提案し、モック→実装→検証→Vol凍結のループで**アイディアが自動的に製品になっていく**工場

## 2. 二本のハーネス

### 守りのハーネス（脱線防止。実装済み）
- `CLAUDE.md` = 全AIの憲法（凍結領域・不変条件・品質ゲート）
- `keydeck-guardian` agent = 変更後の独立点検（規則違反・回帰・凍結領域diff）
- SR起票制（spec_return_log）= 曖昧さを勝手に解釈させない
- 正の三点セット: 設計書（brief/）・モック（mockup/）・盤面JSON（keymaps/）。**実装がズレたら実装を直す**

### 攻めのハーネス（拡張・新型の創出。本書で新設）
- `keydeck-explorer` agent（.claude/agents/）= 使用ログ・既存パーツ・対象アプリを観察し、**改善案と新Surface案を提案書として出す**（実装はしない）
- 提案は `brief/proposals/P-###_<題名>.md` に1件1ファイル（テンプレ: 対象アプリ/画面・課題・提案Surface・流用する既存パーツ・新規が必要な物・受け入れ基準案）
- 昇格条件（ControlDeck future_ideasの5条件を継承）: ①現MVP完成 ②利用場面が1つに定まる ③受け入れ基準を先に書ける ④既存Protocol/スキーマで実装可能か確認済み ⑤10日以内で検証可能な粒度

## 3. 製品化ループ（アイディア→製品の標準工程）

```
観察・着想（explorer/ユーザー）
 → 提案書 P-###（explorer）
 → 裁定（FABLE: 採用/保留/却下。採用ならD番号化）
 → モック（planner-html-mockup。見た目の正を先に確定）
 → 設計5点セット（FABLE。G基準・タスク・禁止表）
 → 実装（Sonnet。凍結仕様のみ）
 → 点検（guardian）→ 検証（Opus。独立再実行）
 → Vol凍結（keymap_<id>_volNN.json＋gitタグ）→ 次のVolへ
```
- どの工程の成果物も**ファイルとしてリポジトリに残る**＝次のAIが文脈ゼロから再開できる

## 4. 新Surface（Inspector・マクロ等）を作るときのagent更新規則

新しい種類の入力面を追加するとき、**コードより先に**次の3点を更新する（これが「担当agentの更新」の答え）:
1. `CLAUDE.md`: そのSurfaceの不変条件を1〜3行追記（例: Inspectorなら「値の確定はHub側検証後のみ」）
2. `keydeck-guardian`: 点検手順に回帰スモーク1行追記（例: 「/inspectorが200・不正値が拒否される」）
3. 正の三点セット: モック＋スキーマ＋設計Dを先に作る（実装は最後）
- guardian/explorerの定義本体は薄く保ち、**知識はCLAUDE.mdと設計書に集約**する（agentが増えても規則が分散しない）
